Terraform statefile(terraform.tfstate):-
1. Terraform must store state about your managed infrastructure and configuration. This state is used by Terraform to map real world resources to your configuration, keep track of metadata, and to improve performance for large infrastructures.
2. This statefile will be modified whenever you execute terraform commands that will impact the real world infra
eg:- terraform apply, terraform destroy
3. It is also used for team collaboration.

Local Statefile:- the file that exists in ur local machine when u execute tf apply
Remote Statefile:- the statefile automatically uploadded to AWS s3 or azure storage. it must be specifies in .tf files then only it will push the code to s3. It avoids dupliaction of resources when multiple people are working

Local statefile Disadvantages:-
“The main disadvantages of Terraform local state are:”
1. No collaboration support
• State is stored locally, so teams cannot share it
• Leads to inconsistencies and duplicate resource creation
2. No state locking
• Multiple users running Terraform simultaneously can corrupt the state
3. Risk of state loss
• If the local file is deleted or corrupted, Terraform loses track of infrastructure
4. Security concerns
• State file may contain sensitive data and is not encrypted by default

✅ Step-by-step flow
1. Read the state file
• What Terraform thinks exists
2. Refresh (very important step)
• Connects to cloud (AWS/Azure)
• Gets actual infrastructure state
• Updates state in memory (not final yet)
3. Compare 3 things
• Your .tf configuration
• Refreshed real infrastructure
• State file
4. Create an execution plan
• What needs to be created/updated/deleted
5. Apply changes to infrastructure
• This is where actual resources are modified
6. Update state file (final step)
• After successful changes, state file is written

🔹 Simple explanation
• Memory = RAM (temporary working space)
• State file (terraform.tfstate) = permanent storage on disk

State Locking in terraform:-
State locking ensures that only one Terraform operation can modify the state at a time, preventing conflicts and corruption
🔴 Why is it Needed?
Terraform state is a single source of truth.
👉 If two people run terraform apply simultaneously:
• Both try to update the state file
• Leads to race conditions
• Results in corrupted or inconsistent infrastructure

🧠 Real-Time Scenario
• You run terraform apply
• Your teammate tries at the same time
👉 Terraform will show:
“Error: state is locked”
This prevents:
• Duplicate resources
• State corruption

🔍 When Locking Happens (Important)
State locking is used for commands that touch the state:
• terraform plan → reads state
• terraform apply → modifies state
• terraform destroy → modifies state
• terraform refresh → updates state

DynamoDB statelocking
“DynamoDB is used by Terraform to maintain a lock record so that only one user can modify the state file at a time.”

👉 Here’s what happens:
1. Terraform tries to run apply
2. It creates a lock entry (row) in DynamoDB table
3. If the lock exists → other users are blocked
4. After completion → lock entry is deleted

🔍 What’s Inside the DynamoDB Table?
• Lock ID
• Operation (apply/plan)
• Who locked it
• Timestamp

Resource Attribute references:-
In Terraform, resource attribute references are how you use values from one resource inside another resource.
Think of it like:
👉 “Take output from Resource A → use it in Resource B”In Terraform, resource attribute references are how you use values from one resource inside another resource.

example:-
# Step 1: Create a VPC
resource "aws_vpc" "main_vpc" {
 cidr_block = "10.0.0.0/16"
}
# Step 2: Create a Subnet inside that VPC
resource "aws_subnet" "subnet1" {
 vpc_id     = aws_vpc.main_vpc.id   # 🔥 Resource Attribute Reference
 cidr_block = "10.0.1.0/24"
}
# Step 3: Create an EC2 Instance inside that Subnet
resource "aws_instance" "web_server" {
 ami           = "ami-12345678"
 instance_type = "t2.micro"
 subnet_id     = aws_subnet.subnet1.id   # 🔥 Another Reference
}
# Step 4: Output the Public IP of the Instance
output "instance_public_ip" {
 value = aws_instance.web_server.public_ip   # 🔥 Attribute Reference
}

Data Sources:-
Think of it like this:
• resource block → creates something
• data block → reads something that already exists
⸻
🔹 Why do we use data sources?
You use data sources when:
• Infrastructure already exists (created manually or by another Terraform project)
• You want to reuse existing values (like VPC IDs, AMI IDs, subnets, etc.)
• You don’t want Terraform to manage (create/delete) that resource

example:-
🧩 Scenario (Very realistic in companies)
• Network team already created a VPC
• Your repo only creates:
• EC2 instance
• Subnet (optional)
👉 You should NOT recreate VPC
👉 You will fetch it using a data source

data.tf:-
# Fetch existing VPC
data "aws_vpc" "existing_vpc" {
 filter {
   name   = "tag:Name"
   values = ["prod-vpc"]
 }
}
# Fetch existing subnet
data "aws_subnet" "existing_subnet" {
 filter {
   name   = "tag:Name"
   values = ["prod-subnet"]
 }
}
# Fetch latest AMI
data "aws_ami" "latest_ami" {
 most_recent = true
 owners      = ["amazon"]
 filter {
   name   = "name"
   values = ["amzn2-ami-hvm-*"]
 }
}

ec2.tf
resource "aws_instance" "my_ec2" {
 ami           = data.aws_ami.latest_ami.id
 instance_type = "t2.micro"
 subnet_id = data.aws_subnet.existing_subnet.id
 tags = {
   Name = "my-ec2"
 }
}

1. terraform init, terraform init -upgrade:-
-> Initializes your project(downloads providers, modules, backends)
-> creates a terraform.lck.hcl with all the providers versions.
-> It will replace the varibles in the terraform configuration files when we are using .tfvars 
When to use:-
-> first time running TF
-> After changing providers/backend

2. terraform plan
-> This will connecy to the cloud provider and fetch the current resources that exists.
-> Its like a dry run which shows what will be created.
-> You can pass variables values just like u do for apply using -var
-> save plan:- terraform plan -out=tfplan
-> terraform plan -refresh=false:- not fetch anything from cloud using local/remote state only

step1:- reads the configuration files
step2:- loads the current state from ur local or remote
step3:- connects to cloud provider and checks the actual infrastructure for any drift
step4:- displays the output what will be created, update or deleted.

NOTE:- why does tf plan connect to cloud when we already have .tfstate that contains the state of the resource in cloud?
verify reality and detect drift (changes made outside Terraform), maybe someone would have manually updated the resource in AWS.

3. terraform apply:- authenticates to AWS and creates the actual resources by comparing the statefile and terrafrom files.
step3.1:- reads the configuration files
step3.2:- loads the current state from ur local or remote
step3.3:- connects to cloud provider and checks the actual infrastructure for any drift
step3.4:- creates actual infrastructure.

1. terraform -help
2. terraform -version

3. terraform fmt:- 
It is used to automatically format Terraform configuration files to a canonical format and style. This helps ensure that your Terraform code is consistent and easy to read, which is especially useful when working in teams.

4. terraform get:-
Note this is usually not required as this is part of the terraform init command
It is used to download and update modules mentioned in your Terraform configuration files. Modules are reusable pieces of Terraform code that can be shared across different projects and teams.

5. terraform validate:-
-> check the syntax and validitate the configuration files in your directory and without actually deploying any resources.
eg:- missing brackets, incorrect parameters names
-> terraform init should be run before this command.

6. terraform taint:-
The terraform taint command is used to mark a resource for recreation during the next terraform apply(only once). This is useful when you want to force the recreation of a resource without making any changes to its configuration. 
eg:- terraform taint <provide>_<resourcetype>.<resourcename>

1. terraform apply -refresh-only:-
This will update the Terraform state to reflect any changes made outside of Terraform, such as manual changes made in the AWS console. This command will only refresh the state without applying any new changes, ensuring that the state is up to date with the actual infrastructure.
